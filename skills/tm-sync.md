# 目的

Gmail から最終 sync 以降のメールを取得し、案件DB の既存案件にひも付けるか新規案件として登録する。確度に応じて自動反映/要確認振り分けを行い、メタ情報DB の最終sync日時を更新する。

このSkillは task-manager の中核。実行頻度は 1 日 1〜数回想定。

# 前提コンテキストの取得

1. `notion-search` で `task-manager 設定` を検索し、該当ページを取得
2. 設定ページ配下のメタ情報DBから config レコードを `notion-fetch` で取得し、以下を読み込む:
   - 案件DB の data_source_id（以降 `DEALS_DSID`）
   - タレントDB の data_source_id（以降 `TALENTS_DSID`）
   - クライアントDB の data_source_id（以降 `CLIENTS_DSID`）
   - 最終sync日時（`LAST_SYNC`、未設定なら 7 日前を仮定）
   - sync対象ラベル（`LABEL_FILTER`、空文字なら無視）
   - バージョン（プロパティ名が日本語か英語か判定。`v1.0-en` 系なら英語名を使用）
3. 設定ページ本文の「ステータス遷移ルール」「AI 判断ルール」を読み、後段の判断で参照する

# 手順

## 1. 辞書・既処理 Set を一括構築（事前ロードフェーズ）

このフェーズで、以降の処理に必要なコンテキストを **一括取得してメモリに展開** する。スレッドごとに Notion を叩く N+1 問題を避けるため、以降のループでは Notion 検索を行わず、ここで作ったメモリ上のデータ構造に対して問い合わせる。

### 1-1. タレント辞書

`notion-search` を呼ぶ。**`query` は最小1文字必須**（空文字は API バリデーションエラー）なので、汎用語を渡す（推奨: `"タレント"` / `"名前"` 等、ヒット率の高い一般名詞）。`data_source_url` パラメータで `TALENTS_DSID` に絞り、ページネーション（`start_cursor`）を辿って全行を取得。

各行から `{名前, [エイリアス], [関連メアド]}` を抽出してメモリ上の `TALENTS` 辞書を作る。

> 注: `notion-search` は意味検索ベースだが、`data_source_url` で範囲を絞ると当該データソース配下のページが返る。MVP の規模（数十〜数百件）なら page_size 25 × 数回のページネーションで全件取得できる。漏れが疑われる場合は別の検索語（複数語）で2回回してマージするフォールバックを Skill 実行中の Claude 自身が判断する。

### 1-2. クライアント辞書

同様に `notion-search` で `CLIENTS_DSID` 配下を全件取得（`query` は `"会社"` 等の汎用語）。各行から `{会社名, [ドメイン], [エイリアス]}` を抽出して `CLIENTS` 辞書を作る。

### 1-3. 案件配列 + Gmail Message ID Set + 逆引きマップ

`notion-search` で `DEALS_DSID` 配下を全件取得（`query` は `"案件"` 等の汎用語、ページネーション必須）。各案件から以下を抽出してメモリ上の `DEALS` 配列に格納:

- ページID、案件名、タレント名、クライアント名、ステータス、納期、最終やり取り日
- `Gmail Message ID履歴` 文字列、既存 AIメモ、重要メモ

並行して以下の 2 つを構築する:

- `PROCESSED_IDS`: **全案件の `Gmail Message ID履歴` を `,` で split して 1 つの Set<string> に展開**。後段の既処理判定はこの Set に対して O(1) で行う。
- `ID_TO_DEAL`: 各 Message ID → 所属案件ページID への辞書（後段マッチングで「このメールはどの案件に属するか」を即引きする）。

> 注: 一括取得は案件数が増えると重い。MVP の想定は数十〜数百件。1000 件超になったら設定ページの sync 対象ラベルで Gmail 側を絞る運用に切り替える。

## 2. Gmail から対象メールを取得

検索クエリを組み立てる:

- 日付: `after:YYYY/MM/DD`（`LAST_SYNC` の日付、スラッシュ区切り。未設定なら 7 日前）
- ラベル: `LABEL_FILTER` が空でなければ `label:<LABEL_FILTER>`
- 例: `after:2026/05/12 label:取引先`

`search_threads(query=..., pageSize=50)` を実行。

取得 0 件なら手順 7 にスキップ。

## 3. 各スレッドを処理（ループ）

各スレッドについて以下を行う:

### 3-1. スレッド本文取得

`get_thread(threadId, messageFormat="FULL_CONTENT")` で取得。

メッセージ数が 5 件以上なら「長スレッド」として後段の AI 解析時に要約を活用（最古 1 通 + 最新 2 通を全文、中間は件名のみで解析）。

### 3-2. 既処理判定（Set 検索）

スレッド内 **全 Message ID** を取得し、`PROCESSED_IDS` Set に対して以下を判定:

- スレッド内最新 Message ID が `PROCESSED_IDS` に含まれる → 既処理（最新メールが既に取り込み済み）。スキップ（`skipped++`）。
- 最新は含まれないが過去のいずれかの Message ID が `PROCESSED_IDS` に含まれる → 既存スレッドに新着あり。スキップせず手順 3-3 で対応案件を `ID_TO_DEAL` から特定して link_existing 候補とする。
- どの Message ID も含まれない → 新規スレッド扱い（マッチング処理へ）。

> Notion 側で `Gmail Message ID履歴 contains` のような文字列部分一致検索はサポートされない / 信頼性がないため、必ず手順 1-3 で構築した Set に対してメモリ上で判定する。

### 3-3. マッチング候補抽出

優先順位は以下:

1. **ID 一致（最優先）**: スレッド内いずれかの Message ID が `PROCESSED_IDS` に含まれていれば、`ID_TO_DEAL` から所属案件を取得して候補にする。これが最も信頼性が高い（機械的識別子の完全一致のため）。
2. **送信ドメイン × タレント名**: 送信元ドメインが `CLIENTS` のドメインに含まれ、かつメール本文中に `TALENTS` のいずれかの名前/エイリアスが出現する → そのタレント × クライアントの既存案件を `DEALS` から検索して候補にする。
3. **件名類似**: 上記でゼロなら、件名（`Re:` / `Fwd:` 除去後）が既存案件名の主要部と類似（編集距離 < 30%）の案件を候補にする。

候補ゼロ → 新規案件候補。

注: 「最古 Message ID = 案件識別子」という考え方は採らない（履歴は最新 20 件のローテーション保持なので、古い ID は失われる可能性がある）。スレッド単位ではなく、保持されている任意の Message ID の一致で同一案件と判定する。

### 3-4. AI 解析（このSkillを実行中の Claude 自身の判断）

スレッド本文・件名・送信者・候補リストを元に以下を判定:

- action: `link_existing` / `create_new` / `needs_review`
- 抽出フィールド:
  - **案件名候補** (新規時のみ): `[クライアント名]_[タレント名]_[案件概要短縮]` 形式
  - **案件概要** (新規時のみ生成、既存案件には変更しない): 30〜60文字で「誰が何をするのか」の不変サマリ。例: 「山田太郎が冬季CMで化粧品Bに出演、年内納品予定」
  - **タレント名**、**クライアント名**、**ステータス推定**、**受注金額**、**納期**
  - **ボール所在**（重要）: 後述の判定指針で4値から選ぶ
  - **次回アクション** (Text): 「先方の見積回答待ち」「契約書ドラフト送付予定」など、次の一手を1行で
  - **次回アクションの期日** (Date): 上記の目安期日。メール中に明示が無くても「○日以内が妥当」を AI が推定可能
  - **AIメモ要約** (1〜2行): 今回のメールで起きた変化
  - **こちらの返信要否**: 「お待ちしております」「ご確認ください」等から判定
  - **is_milestone** (bool) + **milestone_text** (string): このメールで「記録に値する出来事」が起きたか、起きたなら 10〜60 文字で要約
- confidence: 高 / 中 / 低

**マイルストーン判定指針**（ページ本文の「タイムライン」に追記するか否か）:

✅ **マイルストーン = true（追記する）**:
- 金額・納期・スコープが正式に合意された
- 契約締結、契約書サイン、発注確定
- ステータス遷移が起きた（引合→見積中→契約締結→進行中→納品済→入金待ち→完了）
- 納品完了、検収完了
- 請求書送付、入金確認
- 重大な問題・懸念の発生（納期遅延の通知、トラブル発生など）
- クライアントからの正式な意思決定（採用/不採用/保留など）

❌ **マイルストーン = false（追記しない、AIメモのみ）**:
- 単なる進捗確認・状況報告
- 「資料送付しました」「明日電話します」のような連絡調整
- 軽微な質問・回答のやり取り
- 自動配信、CC のみのメール

判断軸は「3ヶ月後に振り返って『この案件で何があったか』を 10 秒で理解できる粒度」。迷ったら false（AIメモには残るので失われない）。

milestone_text のフォーマット: 主語省略可、50字以内、過去形か体言止め。例:
- 「金額¥1.5Mで合意、契約締結へ」
- 「ステータス: 見積中 → 契約締結」
- 「納品完了、検収依頼中」
- 「先方より延期希望、新納期は7/15で調整中」

判定指針（マッチング側）:
- タレント辞書 / クライアント辞書のマッチが両方明確 → 高
- 片方しか確定しない、または金額・期限が文面から特定できない → 中
- 新規ドメイン、タレント未登録、複数案件が混在しているメール → 低（needs_review）

**「ボール所在」判定指針**:
- 直近メールの送信者が自社（事務所）で、本文が「確認します」「対応します」「お送りします」等 → `🔴 自社対応中`（自社が次に動く）
- 直近メールの送信者が自社で「お待ちしております」「ご検討よろしくお願いします」等 → `🟡 クライアント待ち`（先方の返答待ち）
- 直近メールがクライアントからで質問・依頼を含む → `🔴 自社対応中`
- 直近メールがクライアントからで「了解しました」「進めます」等 → `🟢 タレント待ち` または `⚪ 待機・進行中`（実作業者の状況による）
- 撮影/納品など実作業フェーズで進行中 → `🟢 タレント待ち`
- 上記いずれでもない、明確に判定できない → `⚪ 待機・進行中`

**「こちらの返信要否」と「ボール所在」の関係**:
- `🔴 自社対応中` → `こちらの返信要否=true` で強制（必ず連動させる）
- `🟡 クライアント待ち` / `🟢 タレント待ち` / `⚪ 待機・進行中` → `こちらの返信要否=false`（こちらの待ちアクション無し）
- 例外: AI 判断で曖昧なケースは `こちらの返信要否=true` を優先（漏れ防止）

**こちらの返信要否の文面シグナル**:
- 「返信不要」「FYI」「自動配信」「Unsubscribe」が文面に明白 → false 確度高
- 「お返事お待ちしております」「ご検討よろしくお願いします」「ご確認の上」等 → true

### 3-5. Notion 反映

action と confidence に応じて分岐:

> **重要: プロパティ書き込み時の特殊フォーマット**
> - **Date型**: `date:プロパティ名:start`, `date:プロパティ名:end`（任意）, `date:プロパティ名:is_datetime`（0=date / 1=datetime）の3キーで指定
>   - 例: `"date:納期:start": "2026-05-25", "date:納期:is_datetime": 0`
> - **Checkbox型**: `"__YES__"` / `"__NO__"` 文字列で指定（true/false ではない）
>   - 例: `"こちらの返信要否": "__YES__"`
> - **Number型**: JavaScript number で渡す（文字列ではない）
> - **「id」「url」という名前のプロパティ**: `userDefined:` プレフィックスが必要（本Skillの「最新メールへのリンク」は名前が`url`ではないのでそのまま記入可）

#### link_existing + 高
- `notion-update-page` で対象案件のプロパティを更新:
  - 最終やり取り日: スレッド最新メールの日付
  - 最新メールへのリンク: スレッドの Gmail web URL
  - Gmail Message ID履歴: 既存値の末尾に新 Message ID を追加。20 件超過なら古い順から削除
  - AIメモ: 既存内容を保持し `YYYY-MM-DD: <要約>` を末尾追記
  - ステータス: AI解析で推定したステータスが「遷移ルール」上 OK なら更新。逆行する場合は更新しない（重要メモに「[ステータス遷移要確認]」追記）
  - **ボール所在**: AI 判定値で上書き
  - **次回アクション**: AI 判定値で上書き（短い1行、新しい状況を反映）
  - **次回アクションの期日**: AI 判定値で上書き
  - こちらの返信要否: ボール所在と連動（自社対応中=true、それ以外=false）
  - **案件概要**: 既存値が空の場合のみ AI 生成値を入れる。既に入っていれば触らない（不変フィールド）

- **マイルストーン追記** (`is_milestone=true` の時のみ):
  - 別途 `notion-update-page(page_id, command="update_content", content_updates=[...])` を呼ぶ
  - **方針**: 案件ページ本文に既存の「タイムライン」見出しブロックがある前提で、その下に `- YYYY-MM-DD: <milestone_text>` を追記
  - 実装の安全策: 既存本文を `notion-fetch(page_id)` で取得し、「## 📜 タイムライン」の見出し直後を見つけてその下に行を挿入する `update_content` を発行
  - 「タイムライン」セクションが存在しない（古い案件など）場合は `insert_content(position="end")` で末尾に新規セクションを作る:
    ```markdown
    
    ## 📜 タイムライン
    - YYYY-MM-DD: <milestone_text>
    ```

#### link_existing + 中
- 上記 link_existing+高 と同じ更新を実行
- 加えて 重要メモに `[AI要確認 YYYY-MM-DD] <ambiguity の説明>` を追記
- こちらの返信要否=true を強制

#### create_new + 高
- `notion-create-pages` で `DEALS_DSID` 配下に新規案件作成（properties + content を同時指定）:
  - **properties**:
    - 案件名: `[<クライアント名>]_[<タレント名>]_[<案件概要短縮>]` 形式
    - **案件概要**: AI 生成の 30〜60文字サマリ（誰が何をする案件か）
    - タレント名: マッチしたタレント
    - クライアント名: マッチしたクライアント
    - ステータス: AI推定（無ければ `引合`）
    - **ボール所在**: AI 判定
    - 受注金額: 抽出できれば設定（税込なら税抜換算メモを AIメモに）
    - 開始日: スレッド最古メールの日付
    - 納期: 抽出できれば設定
    - 最終やり取り日: スレッド最新メールの日付
    - **次回アクション**: AI 判定（1行）
    - **次回アクションの期日**: AI 判定（メールに記載が無ければ「3日後」を既定）
    - 最新メールへのリンク: Gmail web URL
    - Gmail Message ID履歴: スレッド内の全 Message ID（最新 20 件まで）
    - AIメモ: `YYYY-MM-DD: <要約>`
    - こちらの返信要否: ボール所在と連動
  - **content** (ページ本文、Markdown):
    ```markdown
    ## 🎯 重要事項
    （ここに合意事項・特記事項・注意点などを記入してください。AIは触りません）

    ## 📜 タイムライン
    - YYYY-MM-DD: <初回マイルストーン or 「案件登録」>
    ```
    `is_milestone=true` ならその milestone_text を初回行に、false なら「案件登録」を入れる。

#### create_new + 中
- 上記と同じ更新を実行
- 案件名の先頭に `[要編集]` プレフィックス
- 重要メモに `[AI要確認 YYYY-MM-DD] <ambiguity の説明>` を追記
- こちらの返信要否=true を強制

#### needs_review / 低
- Notion には書き込まない
- 要確認リスト（メモリ上の配列）に以下を退避: `{threadId, 件名, 送信者, 日時, 想定アクション, ambiguity 理由}`

### 3-6. 複数案件が混在するメール

1 スレッド内で 2 件以上の案件が議論されていると AI が判定した場合:
- 各案件ごとに分割して上記処理を実行
- AIメモに `1スレッド複数案件として分割登録` を明記

### 3-7. 添付ファイル

`get_thread` で添付情報があればファイル名のみ取得し、AIメモに `添付: <filename1>, <filename2>` 形式で記録（実体は取得しない）。

## 4. ステータス遷移の検証

更新時にステータスを変更する場合、設定ページの遷移ルールを参照:

許可される遷移のみ反映。逆行（例: `納品済` → `進行中`）は反映せず、重要メモに `[ステータス遷移要確認] 現状=<old> 推定=<new>` を追記。

「中断」経由の再開は許可。

## 5. Gmail Message ID 履歴のローテーション

各更新時に Message ID履歴を更新するロジック:

1. 既存履歴を `,` 区切りでパース
2. 新規 Message ID を末尾に追加
3. 重複削除（既存があれば後発を残す）
4. 21 件以上になったら先頭から削除して 20 件に揃える
5. `,` 区切りで結合して保存

## 6. メタ情報DB の最終sync日時を更新

config レコードを `notion-update-page` で現在日時に更新。

タイムゾーンは Asia/Tokyo（friend は日本在住）。

> 注: 最終sync日時は DATE 型のため日付単位の精度。同日に 2 回 sync を回すと Gmail 検索 `after:YYYY/MM/DD` で 1 回目処理済みのメールも再取得対象になるが、手順 1-3 で構築した `PROCESSED_IDS` Set による既処理判定（手順 3-2）で重複は自動的にスキップされるため問題ない。

# 判断指針 / エッジケース

- **新規ドメイン**: クライアントDBに存在しないドメインから来たメールは confidence 低。要確認に退避。
- **複数タレントが文面に登場**: メインタレントを特定できなければ needs_review。
- **税込価格**: 「¥100,000（税込）」を見つけたら税抜き換算（÷1.1）を受注金額に入れる。**小数点以下は円単位で切り捨て（floor）して整数化**（例: ¥100,000 税込 → ¥90,909 税抜）。AIメモに「税込¥100,000 → 税抜¥90,909（切り捨て）」と記録。
- **曖昧期限**: 「来週中」「月末まで」は具体日付に変換（今日基準）。「なるべく早く」は納期に入れず AIメモに記録。
- **返信不要明白**: 「FYI」「ご報告まで」「自動配信」「@noreply」「unsubscribe」を含む → 返信要否=false で高確度。
- **長スレッド（5通以上）**: 最古1通 + 最新2通を全文、中間は件名のみで AI 解析。AIメモに「長スレッド要約」と明記。
- **ステータス逆行**: ルール上 NG な遷移は実行せず、重要メモに記録。
- **添付ファイル**: ファイル名のみ AIメモに記録。実体取得はしない。
- **エラー時のリトライ**: Notion API 失敗は 1 度だけリトライ。それでも失敗したら要確認に退避して次へ。
- **スレッドID と Message ID の使い分け**: 既処理判定は Message ID 履歴で行う（スレッドに新着が来たら sync 対象）。

# 完了条件

- 取得した全スレッドが処理済み（更新 / 新規 / スキップ / 要確認 のいずれかに分類）
- メタ情報DB の最終sync日時が現在時刻に更新されている
- 要確認件数を含むレポートがユーザーに提示されている
- Notion に書き込んだ全案件で Message ID 履歴が 20 件以内
