# 目的

過去 7 日間の案件動向を集計し、friend が経営判断に使える形でサマリを提示する。動き（新規/更新/完了）、現状（進行中/入金待ち/要返信/納期超過）、注目案件、来週のフォーカスを Markdown で出力。

# 前提コンテキストの取得

> 🚨 **セットアップ未完了の検知**: 下記手順 1〜3 のいずれかが 0件 / 見つからない場合は **即座に中断**し、「❌ task-manager のセットアップが完了していません。Claude Desktop で `/tm-setup` を実行してください」とユーザーに返す。推測でデフォルト値を埋めて続行しない。

1. `notion-search(query="task-manager 設定", page_size=5)` で設定ページの page_id を取得（0件なら中断）
2. `notion-fetch` で設定ページ → メタ情報DB の `data_source_id` を抽出（`META_DSID`）
3. `notion-search(query="メタ情報DB", data_source_url="collection://"+META_DSID, page_size=5)` で config レコード（通常1件）を特定し、`notion-fetch` で:
   - 案件DB の data_source_id（`DEALS_DSID`）

# 手順

## 1. 案件DB の全件取得（一括取得 → メモリで集計）

Notion Connector の `notion-search` は意味検索ベースで、Boolean / 日付範囲 / `NOT IN` 等の構造化フィルタはサポートされない / 信頼性が低い。また `query` は **最小1文字必須**（空文字は API エラー）、かつ **汎用語ではほぼヒットせず、データソース名を渡すと全件返る** という仕様。`notion-fetch` は ID 単位取得のみ。そのため次の手順を取る:

1. `notion-search(query="案件DB", data_source_url="collection://"+DEALS_DSID, page_size=25)` で案件DB全件取得。`start_cursor` でページネーション対応。中断・完了案件も含めて取得する（集計対象は手順 2 以降でメモリ上の条件で絞る）。
2. 取得した全案件を以降の手順で **このSkill実行中の Claude 自身がメモリ上で集計** する。

今日の日付を `TODAY`、7 日前を `WEEK_AGO` とする（タイムゾーン Asia/Tokyo）。

> パフォーマンス注記: この一括取得は案件数が増えると重い。MVP の想定は数十〜数百件のため許容範囲。100 件を超え始めたら出力末尾に「※ 案件数が多くなってきました。集計が遅い場合は Notion の View（フィルタ済みビュー）を直接見る運用への切り替えを検討してください」と添える。

## 2. 動きの集計（メモリ上のフィルタリング）

### 新規作成案件
- `開始日 >= WEEK_AGO` の案件数

### 更新があった案件 (※近似集計)
- `最終やり取り日 >= WEEK_AGO` かつ `開始日 < WEEK_AGO` の案件数（新規以外で動きがあったもの）
- **注**: ステータス変化の正確な履歴は保持していないため、本項は最終やり取り日基準の近似値（出力時の表示にも「※近似」を併記）。

### 完了になった案件
- `ステータス = '完了'` かつ `最終やり取り日 >= WEEK_AGO` の案件
- これらの受注金額の合計も算出

## 3. 現状の集計（メモリ上のフィルタリング）

- 進行中: `ステータス = '進行中'` の件数
- 入金待ち: `ステータス = '入金待ち'` の件数と受注金額合計
- **ボール別内訳** (未完了案件のみ):
  - 🔴 自社対応中: `ボール所在 = '🔴 自社対応中'`
  - 🟡 クライアント待ち: `ボール所在 = '🟡 クライアント待ち'`
  - 🟢 タレント待ち: `ボール所在 = '🟢 タレント待ち'`
  - ⚪ 待機・進行中: `ボール所在 = '⚪ 待機・進行中'`
- 要返信: `ボール所在 = '🔴 自社対応中'` または `こちらの返信要否 = true`（旧スキーマ互換）の件数
- 納期超過: `納期 < TODAY` かつ `ステータス NOT IN ('完了', '中断', '入金待ち', '納品済')` の件数とリスト
- **次回アクション期日超過**: `次回アクションの期日 < TODAY` かつ `ステータス NOT IN ('完了', '中断')` の件数とリスト

## 4. 注目案件の抽出

以下のいずれかに該当する案件を「注目」として 3 件まで選出:

- 重要メモに `[AI要確認 ...]` が含まれる
- 納期が `TODAY` から 7 日以内
- 受注金額が大きい上位（基準: 上位20%）

それぞれに 1 行コメント（要点）を添える。

## 5. 来週のフォーカス提案

AI（このSkill実行中のClaude自身）が、上記の集計と注目案件を見て 2〜4 個の提案を生成:

- 「Xさんの××案件は納期間近、要連絡」
- 「Y社の入金が予定日を過ぎている、リマインド推奨」
- 「Z案件の確度が中のまま放置されている、確認推奨」

具体的な案件名を含めること。

## 6. 出力 Markdown

```
📊 週次レポート (YYYY-MM-DD 〜 YYYY-MM-DD)

※ 本レポートはステータス変化の履歴を保持していないため、「更新があった案件」は最終やり取り日基準の近似値です。

## 動き
- 🆕 新規案件: <N>件
- 🔄 更新があった案件 (※近似): <N>件
- ✅ 完了: <N>件 (合計 ¥<X,XXX>)

## 現状
- 進行中: <N>件
- 入金待ち: <N>件 (合計 ¥<X,XXX>)

## ボール所在内訳 (未完了案件)
- 🔴 自社対応中: <N>件 ← ここが直ちに動くべき案件
- 🟡 クライアント待ち: <N>件
- 🟢 タレント待ち: <N>件
- ⚪ 待機・進行中: <N>件

## 期日・納期アラート
- 🔥 次回アクション期日超過: <N>件
  - <案件名1> 期日: YYYY-MM-DD (-N日) — 次回アクション: <内容>
- ❗ 納期超過: <N>件
  - <案件名1> 納期: YYYY-MM-DD (-N日)
  - <案件名2> ...

## 注目案件
1. <案件名> — <要点>
2. <案件名> — <要点>
3. <案件名> — <要点>

## 来週のフォーカス
- <提案1>
- <提案2>
- <提案3>
```

「動き」が全部ゼロの週は「動きなし」を明示。納期超過がゼロの場合はリストごと省略。

# 出力フォーマット

手順 6 の通り。数値が 0 でも該当項目は表示（ゼロを明示することに価値がある）。

# 判断指針 / エッジケース

- ステータス変化の正確な履歴は保持していないため、「更新があった案件」は最終やり取り日基準の近似値。出力テンプレートの冒頭に免責事項として記載し、該当行にも「(※近似)」を併記する。
- 受注金額が空の案件は集計から除外（金額ベースの合計には含めない）。
- 「税抜換算」が AIメモにある場合は税抜金額で集計。
- 出力する金額は ¥ 記号 + 3桁カンマ区切り。

# 完了条件

- 動き / 現状 / 注目 / フォーカス の 4 セクションが揃っている
- 入金待ち合計金額が明示されている
- 来週のフォーカスが具体的な案件名を含む
